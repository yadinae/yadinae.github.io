摘要

本文记录在 Linux VPS 上配置 gog（Google Workspace CLI 工具）的完整过程，包括安装、OAuth 凭证创建、授权流程以及遇到的问题和解决方案。通过本文，你将学会如何在无图形界面的服务器上完成 Google OAuth 授权，并使用 gog 管理 Gmail、Calendar、Drive 等 Google 服务。

为什么需要 gog？

在自动化工作流中，经常需要与 Google 服务交互：

  • 读取 Gmail 邮件触发自动化任务
  • 管理 Calendar 日程安排
  • 操作 Drive 文件
  • 同步 Contacts 联系人

gog 是一个强大的命令行工具，支持以下 Google 服务：

  • Gmail - 邮件收发、搜索、标签管理
  • Calendar - 日程创建、查询、修改
  • Drive - 文件上传、下载、分享
  • Contacts - 联系人管理
  • Docs/Sheets/Slides - 文档编辑
  • Chat/Classroom/Tasks - 协作工具

gog 简介

gog 是一个用 Go 编写的 Google Workspace CLI 工具，特点包括：

  • 🚀 快速 - 原生二进制，启动迅速
  • 🔐 安全 - OAuth 2.0 认证，token 加密存储
  • 📦 全能 - 支持 12+ Google 服务
  • 🎯 精准 - 命令行参数精确控制

项目地址: https://github.com/lucasepe/gog

问题 2：授权端口变化

现象：每次启动授权流程，回调端口都不同（40147 → 38111 → 35581 → 46263 → 41609）

原因：gog 使用随机可用端口启动临时 HTTP 服务器接收 OAuth 回调

解决方案：

  1. 启动授权命令后，立即从日志中获取当前端口的授权 URL
  2. 完成授权后，使用对应端口的回调 URL 发送 curl 请求
  3. 确保授权流程在 30 秒内完成（避免进程超时）

问题 3：授权进程超时

现象：授权命令在 60 秒后超时退出

原因：exec 工具默认超时限制

解决方案：使用 nohup 后台运行

经验总结

教训

  1. Linux 服务器缺少 keyring 后端 - 桌面系统通常有 GNOME Keyring 或 KWallet，但服务器没有
  2. 必须设置 GOG_KEYRING_PASSWORD - 这是 gog 在无 keyring 环境下的降级方案
  3. 授权流程要快速 - 临时 HTTP 服务器会超时退出
  4. 端口是随机的 - 每次授权前要从日志中获取正确的端口

最佳实践

  1. 使用 nohup 后台运行授权命令 - 避免超时
  2. 提前准备好 Google Cloud 项目 - 包括 API 启用和 OAuth 配置
  3. 记录 Client ID 和 Secret - 方便后续重新配置
  4. 将密码添加到环境变量 - 避免每次手动输入
  5. 测试所有服务 - 确保授权范围正确

安全建议

  1. 限制 OAuth 范围 - 只授权必要的服务
  2. 使用测试用户 - 生产环境前先用测试账户验证
  3. 保护 Client Secret - 不要提交到版本控制
  4. 定期轮换凭证 - 定期更新 OAuth 凭证
