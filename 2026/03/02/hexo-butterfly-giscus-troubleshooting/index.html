⚠️ 核心问题定位

YAML 规范不允许同一层级出现重复的映射键（duplicated mapping key）。
当配置文件存在重复键时，解析器会：

  • 报错退出（严格模式）
  • 使用后一个值覆盖前一个（宽松模式）
  • 静默失败（Hexo 的行为）

本案例中，Hexo 选择了第三种方式，导致配置”看似成功”但实际未生效。

根因分析

为什么会出现重复配置？

Butterfly 主题的 _config.yml 文件中，除了实际配置块外，还包含一些配置示例模板。这些模板用于说明配置格式，但不应该被激活。

在我之前的编辑过程中，不小心保留了模板中的 giscus: 配置块，导致 YAML 解析器遇到重复键。

为什么 YAML 解析失败但没有报错？

📝 根据经验：Hexo 配置解析失败时通常不会阻断生成流程，而是使用默认配置继续生成。

这就是为什么：

  • hexo generate 没有报错
  • 部署成功
  • 但配置不生效

验证方法：检查生成的 HTML 是否包含预期内容，而不是仅依赖 hexo generate 的退出码。

经验总结

教训

  1. YAML 配置要格外小心

    • 重复键会导致解析失败
    • 静默失败比报错更危险

  2. 不要相信”没有报错”

    • 没有报错 ≠ 配置正确
    • 一定要验证生成的结果

  3. 备份！备份！备份！

    • 编辑配置文件前必备份
    • 使用带日期的备份文件名

  4. 使用工具验证

    • YAML 验证工具
    • grep 检查重复键
    • curl 验证远程结果
